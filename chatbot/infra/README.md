# Chatbot Infra

Terraform for the chatbot's *own* hosting ā€” deliberately separate from
`environments/{dev,qa,prod}`, which model AzureLearning workload
infrastructure through the JSON-model pattern the chatbot itself writes to.
This is hand-written platform tooling, applied directly by a human, not
generated per-request.

## What this provisions

Deploys into the **existing** `azure-learning-dev` resource group (the
`primary` entry in `models/dev/resource-group.json`, owned by
`environments/dev`'s own Terraform state) ā€” referenced here via a `data
"azurerm_resource_group"` block, never created or owned by this state file.

| Resource | Module | Choice | Why |
|---|---|---|---|
| Key Vault | `modules/key_vault` | Standard SKU, RBAC auth | Holds `anthropic-api-key` and `github-token` |
| Container Registry | `modules/container_registry` | Basic SKU, no admin user | Cheapest tier; pull is via managed identity |
| Log Analytics workspace | `modules/log_analytics_workspace` | PerGB2018, 30-day retention | Platform minimum, required by Container Apps |
| Container Apps environment | `modules/container_app_environment` | Consumption only, no VNET | No idle cost beyond Log Analytics ingestion |
| Container App (backend) | `modules/container_app` | `min_replicas = 0`, 0.25 vCPU / 0.5Gi | Scales to zero ā€” $0 while idle, few-second cold start |
| Static Web App (frontend) | `modules/static_web_app` | Free tier | $0 hosting for the Next.js app |

Each of these lives as a standalone module under `modules/` ā€” the same
pattern as `modules/resource_group`, `modules/storage_account`, and
`modules/virtual_network` used elsewhere in this repo ā€” not inlined into
`chatbot/infra/main.tf`. That's deliberate: they're generic building
blocks (any future workload can reuse `modules/container_app`, not just
this chatbot), and it's what makes them the natural next candidates for
their own `models/schema/*.schema.json` + `environments/dev` wiring if any
of these ever need to be creatable through the chatbot's own UI, the same
way `resource-group`/`storage-account`/`virtual-network` are today.
`chatbot/infra/main.tf` itself stays a thin composition root wiring those
modules together plus the chatbot-specific glue (secrets, role
assignments, the generated backend API key) that doesn't belong in a
reusable module.

## Prerequisites

1. A GitHub token (`github_token` var) with `contents:write` +
   `pull_requests:write` on this repo ā€” same requirement the local backend
   already has via `gh auth login`, just as a token instead of an
   interactive session. Fine-grained PAT scoped to this one repo, or a
   GitHub App installation token.
2. An Anthropic API key (`anthropic_api_key` var).
3. Docker, to build and push the backend image (see
   `../backend/Dockerfile`).

## Deploy

```bash
cd chatbot/infra
terraform init -backend-config=backend.hcl
terraform plan \
  -var="subscription_id=<sub-id>" \
  -var="anthropic_api_key=$ANTHROPIC_API_KEY" \
  -var="github_token=$GH_TOKEN"
terraform apply ...same vars...
```

Then build and push the backend image (run from the repo root, not
`chatbot/backend/`, since the Dockerfile needs the whole repo as build
context):

```bash
ACR=$(terraform output -raw container_registry_login_server)
az acr login --name "${ACR%%.*}"
docker build -f chatbot/backend/Dockerfile -t "$ACR/chatbot-backend:latest" .
docker push "$ACR/chatbot-backend:latest"
```

The Container App pulls `:latest` by default (`container_image_tag`
variable) ā€” pushing a new image doesn't itself trigger a new revision;
either re-run `terraform apply` after changing `container_image_tag`, or
add a CI step that does `az containerapp update --image ...`.

## Frontend deploy ā€” known gap

This Terraform only provisions the Static Web App *shell* and outputs its
deployment token (`static_web_app_api_key`). Actually publishing the
Next.js build still needs a build/deploy step, and there's a real caveat
worth flagging before wiring it up: this frontend uses App Router server
routes (`src/app/api/**`) to keep `ANTHROPIC_API_KEY`/`BACKEND_API_KEY`
off the client ā€” Azure Static Web Apps' Next.js "hybrid" support runs
those via a managed Azure Functions integration, which has narrower
Node API coverage than a normal Next.js server. Test the BFF routes
specifically after the first real deploy before trusting this in
production; a plain Azure Container App (same pattern as the backend) is
the fallback if SWA's Next.js support turns out to be too limited.

## Secrets flow

`anthropic_api_key` and `github_token` are written to Key Vault by
Terraform, then referenced directly from the Container App's `secret`
blocks via `key_vault_secret_id` + the container app's own managed
identity (`Key Vault Secrets User` role) ā€” the key material is never
written to a container env file or image layer. The `x-api-key` the
frontend must call the backend with is generated by Terraform
(`random_password.backend_api_key`) and exposed only via the sensitive
`backend_api_key` output ā€” read it with `terraform output -raw
backend_api_key` and put it in the frontend's `BACKEND_API_KEY` env var.
